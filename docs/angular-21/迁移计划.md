# Angular 21 现代化迁移计划

## 概述

将项目从 Angular 14 架构迁移到 Angular 21 现代化架构。

**原则**:
- ✅ 每步独立、可回滚
- ✅ 不破坏现有功能
- ✅ 可以分批提交
- ✅ 渐进式改进

---

## 迁移路线图

```
[1. 移除 standalone: false] → [2. 新控制流语法] → [3. Signals] → [4. 函数式 API] → [5. Standalone] → [6. Zoneless]
   (最简单)              (中等)          (中等)       (中等)         (复杂)         (可选)
```

---

## 阶段 1: 移除 `standalone: false` ✅ 推荐

**优先级**: 🔴 最高
**难度**: ⭐ 极简单
**工作量**: 5 分钟
**风险**: 极低

### 为什么从这个开始？

1. **最简单** - 只需删除代码
2. **影响最大** - 265 个文件
3. **无风险** - Angular 20+ 默认就是 standalone
4. **立即见效** - 符合 Angular 21 标准

### 执行步骤

```bash
# 1. 使用自动化脚本删除所有 standalone: false
find src/app -name "*.ts" -type f ! -name "*.spec.ts" -exec sed -i '/standalone: false,/d' {} \;

# 2. 验证构建
npm run build

# 3. 提交
git add -A
git commit -m "refactor: remove standalone: false from all components"
```

### 影响范围
- **265 个文件**
- **零功能变更**
- **完全向后兼容**

## 阶段 2: 新控制流语法 (@if, @for)

**优先级**: 🟡 高
**难度**: ⭐⭐ 简单
**工作量**: 1-2 小时
**风险**: 低

### 为什么第二优先？

1. **性能提升** - 新控制流更快
2. **可读性提升** - 代码更清晰
3. **工具支持** - Angular CLI 可以自动迁移

### 执行步骤

#### 2.1 使用自动迁移（推荐）

```bash
# Angular CLI 提供了自动迁移
ng update @angular/core --name migrate-control-flow
```

#### 2.2 手动迁移（如果自动迁移失败）

**示例**:
```html
<!-- 修改前 -->
<div *ngIf="isVisible">Content</div>
<div *ngFor="let item of items">{{ item.name }}</div>

<!-- 修改后 -->
@if (isVisible) {
  <div>Content</div>
}
@for (item of items; track item.id) {
  <div>{{ item.name }}</div>
}
```

### 迁移批次

| 批次 | 文件数 | 优先级 | 模块 |
|------|--------|--------|------|
| 1 | 50 | 高 | 常用组件 (user-management, projects) |
| 2 | 100 | 中 | project-map, preferences |
| 3 | 200+ | 低 | cartography, 其他 |

### 验证

```bash
# 每批迁移后
npm run build
npm test
```

## 阶段 3: 引入 Signals

**优先级**: 🟢 中
**难度**: ⭐⭐⭐ 中等
**工作量**: 4-8 小时
**风险**: 中

### 策略

**不要一次性全部迁移** - 选择影响最大的部分先迁移：

#### 3.1 第一批：列表组件 (2小时)

**目标文件**:
- `src/app/components/user-management/user-management.component.ts`
- `src/app/components/projects/projects.component.ts`
- `src/app/components/snapshots/list-of-snapshots/list-of-snapshots.component.ts`

**迁移模式**:
```typescript
// 修改前
items: Item[] = [];
filteredItems = Item[] = [];

filter() {
  this.filteredItems = this.items.filter(...);
}

// 修改后
items = signal<Item[]>([]);
filteredItems = computed(() =>
  this.items().filter(...)
);
```

#### 3.2 第二批：表单组件 (2-3小时)

**目标文件**:
- `src/app/components/login/login.component.ts`
- `src/app/components/settings/console/console.component.ts`

#### 3.3 第三批：其他组件 (2-3小时)

## 阶段 4: 函数式 API (input/output)

**优先级**: 🟢 中
**难度**: ⭐⭐⭐ 中等
**工作量**: 3-5 小时
**风险**: 中

### 执行步骤

#### 4.1 迁移 input() 函数

**示例**:
```typescript
// 修改前
@Input() items: Item[] = [];
@Input() disabled: boolean = false;

// 修改后
items = input<Item[]>([]);
disabled = input<boolean>(false);

// 读取值 (必需的变更)
const items = this.items();  // 添加 ()
const disabled = this.disabled();
```

#### 4.2 迁移 output() 函数

**示例**:
```typescript
// 修改前
@Output() saved = new EventEmitter<boolean>();

// 修改后
saved = output<boolean>();

// 发送事件 (无需变更)
this.saved.emit(true);
```

### 迁移批次

| 批次 | 文件数 | 组件类型 |
|------|--------|---------|
| 1 | 20 | 简单组件 (5个以内 input/output) |
| 2 | 40 | 中等组件 (5-10个) |
| 3 | 45+ | 复杂组件 (10+个) |

## 阶段 5: 迁移到 Standalone 组件

**优先级**: 🟢 中低
**难度**: ⭐⭐⭐⭐⭐ 复杂
**工作量**: 15-20 小时
**风险**: 高

### 为什么放在后面？

1. **需要大量重构**
2. **可能破坏现有功能**
3. **需要充分测试**
4. **可以逐步进行**

### 策略：混合模式

**不要一次性全部迁移** - 新组件用 standalone，旧组件保持 NgModule：

```typescript
// ✅ 允许混合
@Component({
  standalone: true,  // 新组件
  imports: [CommonModule, MatButtonModule]
})
export class NewComponent { }

@NgModule({
  declarations: [OldComponent]  // 旧组件保持不变
})
export class FeatureModule { }
```

### 执行步骤

#### 5.1 创建新的 Standalone 组件

**示例**: 重新创建一个简单组件作为 standalone

#### 5.2 迁移一个完整模块

**推荐从简单模块开始**:
1. `confirmation-dialog.component.ts` (已经是 standalone)
2. `tool-details-dialog.component.ts` (已经是 standalone)
3. 选择一个新的小型模块

#### 5.3 逐步迁移大型模块

**最后一个迁移**: `app.module.ts`
